## 프로젝트 선정 이유
스프링 부트를 실행하면 "Tomcat initialized with port(s) : 8080"라는 문구가 뜨면서 포트가 열리는데 네트워크 수업에서 TCP/IP 프로토콜 등을 배우며 스프링 부트가 내부적으로 어떻게 포트를 여는지에 대한 이해를 높이기 위해서

### Tomcat
- HTTP와 WebSocket을 처리하기 위한 TCP 기반의 WAS
- 과거에는 Tomcat을 컴퓨터에 설치하고 자바 코드를 Tomcat에 배포해야 했음 → Spring Boot는 Tomcat을 Java 라이브러리(jar) 형태로 서버 내부에 내장하고 있음
- Spring Boot가 실행될 때 "spring-boot-autoconfigure" 모듈이 내장 톰캣 라이브러리가 프로젝트에 존재하는지 확인하고 존재한다면 Tomcat 서버를 자바 객체(Tomcat 클래스)로 직접 생성하는 자동 설정 코드 실행

### ServerSocket
- 자바(JDK)가 태생적으로 가지고 있는 표준 네트워크 통신용 클래스
- 특정 포트 번호를 쥐고, 그 포트로 들어오는 외부 연결 요청을 기다리는 감시원 역할

## 1단계. 8080 포트 열고 브라우저 접속만 확인하기
- **목표**: 서버 소켓을 생성하여 포트를 개설하고, 최초의 TCP 연결을 수립
- **핵심 메커니즘** : 'serverSocket.accept()'는 호출 시 클라이언트의 요청이 올 때까지 스레드가 멈추는 Blocking 상태가 된다. 브라우저가 'localhost:8080'으로 접속하면 OS 커널에서 TCP 3-Way Handshake가 완료되고, 대기 큐에서 연결 정보를 꺼내와 클라이언트 전용 'Socket' 객체를 반환한다.
- **한계점** : 연결(Transport 레이어)은 성공했으나 애플리케이션 레이어의 HTTP 요청 메시지를 읽지 못해, 브라우저는 응답을 받지 못하고 무한 로딩 상태에 빠짐 -> 2단계에서 해결

## 2단계. HTTP 요청(Request) 메시지 파싱 및 콘솔 출력
- **목표** : 클라이언트(브라우저)가 보낸 애플리케이션 레이어의 데이터(HTTP 요청 프로토콜)를 자바 스트림을 통해 읽어와 콘솔에 출력
- **핵심 메커니즘**
  1. 'clientSocket.getInputStream()'을 통해 네트워크로 들어오는 날것의 바이트 데이터를 수집
  2. 'InputStreamReader'를 사용하여 바이트 데이터를 인간이 읽을 수 있는 문자 형태로 통역
  3. 'BufferedReader'의 'readLine()'을 활용하여 문자 데이터를 버퍼에 모았다가 엔터(줄바꿈) 단위로 읽어옴
- **한계점**: 브라우저가 보낸 요청은 읽었으나, 서버 측에서 아무런 응답 메시지를 돌려주지 않아 브라우저는 여전히 흰 화면에서 무한 로딩 상태를 유지 → 3단계에서 해결

## 3단계. HTTP 응답(Response) 송신 및 브라우저 화면 출력
- **목표** : 요청을 보낸 브라우저에게 정식 HTTP 응답 규격에 맞는 메시지를 전송하여 무한 로딩을 해결하고, 브라우저 화면에 첫 HTML 텍스트를 성공적으로 렌더링한다.
- **핵심 메커니즘**
  1. 'clientSocket.getOutputStream()' 기반의 DataOutputStream을 개설하여 클라이언트 방향의 출력 스트림을 확보한다.
  2. HTTP 프로토콜 응답 규격(Status Line, Response Header, Blank Line, Response Body)을 순서대로 바이트 단위로 전송한다.
  3. 'Cotent-Type과 Content-Length 헤더를 정확히 명시하여 브라우저가 데이터를 안전하게 파싱할 수 있도록 한다.
- **한계점**
  1. 클라이언트에게 응답을 1회 송신한 후 자바의 main 스레드 흐름이 끝나 서버가 즉시 종료됨 → 4단계에서 무한루프로 해결
  2. 하나의 스레드로만 실행되므로, 앞선 클라이언트의 요청-응답 사이클이 완료되기 전까지는 다른 클라이언트가 접속하더라도 대기 큐에서 대기해야 하는 동시성 문제가 존재 → 5단계에서 Multi-Thread 도입으로 해결

## 4단계. while(true) 무한 루프 기반의 지속 가능한 서버 구동
- **목표** : 단발성으로 종료되던 기존 서버의 한계를 극복하고, 24시간 내내 클라이언트의 요청을 지속적으로 수신할 수 있는 루프 구조 확립
- **핵심 메커니즘**
  1. while(true) 제어문을 도입하여 accept()부터 요청 파싱, 응답 송신에 이르는 전 과정을 반복 구동 시킴
  2. serverSocket.accept()의 Blocking 특성 덕분에 무한 루프가 돌더라도 클라이언트의 요청이 없을 때는 CPU 자원을 낭비하지 않고 대기 상태를 유지함
  3. 하나의 클라이언트 응답 사이클이 끝나면 clientSocket.close()를 명시적으로 호출하여 소켓 자원을 반환하고, 다음 루프 회차의 accept()가 정상적으로 다음 클라이언트를 맞이할 수 있도록 처리
- **한계점** : 지속적인 요청 처리를 가능해졌으나 여전히 싱글 스레드로 동작 → 5단계에서 해결
